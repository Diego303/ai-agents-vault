Resumen

RAG es una arquitectura que combina la recuperación de información (retrieval) con la generación de texto de un LLM, permitiendo respuestas fundamentadas en datos reales y reduciendo alucinaciones en un 40-60% según estudios recientes[1]. ^resumen-rag

1. Qué es y por qué importa

RAG (Retrieval-Augmented Generation) fue introducido por Meta AI en 2020 como respuesta al problema fundamental de los LLMs: generan texto fluido pero no pueden acceder a información actualizada ni verificar sus propias afirmaciones.

El concepto es simple: antes de que el LLM genere una respuesta, buscamos información relevante en una base de conocimiento y la inyectamos en el prompt como contexto.

Cuándo usar RAG vs Fine-tuning
  • RAG: Cuando necesitas datos actualizados, fuentes verificables, o el conocimiento cambia frecuentemente.
  • Fine-tuning: Cuando necesitas cambiar el comportamiento o estilo del modelo.
  • Ambos: Se pueden combinar. Ver rag-vs-fine-tuning para un análisis detallado.

El pipeline consta de dos fases principales que se detallan en rag-pipeline-completo:

  1. Indexación (offline): Documentos → chunking-strategies|chunks → embeddings → vector-databases|base vectorial.
  2. Recuperación + Generación (online): Query → retrieval → reranking → generación.

3. Implementación básica

Ver implementación en Python con LangChain 
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.chains import RetrievalQA
from langchain.llms import ChatOpenAI

# Indexación
embeddings = OpenAIEmbeddings()
vectorstore = Chroma.from_documents(documents, embeddings)

# Retrieval + Generation
qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-4"),
    retriever=vectorstore.as_retriever(search_kwargs={"k": 5}),
    return_source_documents=True
)
result = qa_chain.invoke({"query": "¿Cómo funciona RAG?"})

4. Variantes avanzadas

Variante Descripción Cuándo usar
Corrective RAG (CRAG) Evalúa relevancia del contexto recuperado antes de generar. Cuando la calidad del retrieval es variable.
Self-RAG El LLM decide dinámicamente si necesita retrieval o no. Queries mixtas (algunas necesitan contexto externo, otras no).
Agentic RAG Un agente orquesta múltiples retrievals con herramientas. Queries complejas multi-hop que requieren razonamiento.
Graph RAG Usa bases de datos de grafos (Knowledge Graphs). Extracción de relaciones complejas entre entidades.
Limitación crítica

RAG no soluciona el problema de razonamiento. Si el LLM no puede razonar correctamente sobre el contexto recuperado, la respuesta seguirá siendo incorrecta aunque los documentos sean perfectos. Ver razonamiento-llm para más contexto.
